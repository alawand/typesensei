  Monkeytype is the bar for feel — instant start, smooth caret, keypress sounds,

  ZType (the space shooter) has a trap I designed around. It makes you type
  isolated words fast → trains hunt-and-peck, the wrong skill. So the central
  design rule for Typesensei is:

  ▎ The game reacts to your typing. Your typing never bends to the game.

⏺ Here's the stack explained from a systems-engineer's point of view, not a
  web-dev's. A browser app is assembled from ~7 layers. For each I'll tell you
  what it is (with an analogy you'll know), the options, and my pick.

  ---
  1. Language — TypeScript vs JavaScript

  JavaScript is the only language a browser runs natively. TypeScript is
  JavaScript plus a real static type system; it compiles down to JS. Same
  relationship as Kotlin→JVM bytecode.

  For you specifically — you're building a stateful typing engine with a strict
  error state machine, and you value correctness — types pay for themselves
  immediately.
  → TypeScript. Not really debatable.

  2. UI framework — React / Svelte / Solid / none

  The browser exposes a low-level tree-of-UI-elements API (the DOM) — think raw
  syscalls for drawing widgets. A framework lets you instead declare "screen =
  function(state)" and it handles diffing and updating the actual pixels. It's
  the front-end analog of Spring/Django: structure + plumbing so you don't
  hand-roll it.

  - React — industry default. Largest ecosystem, the most documentation and
  example code, and — relevant to us — by far the most training data, so AI
  assistance (me) is most reliable here. Slightly verbose.
  - Svelte — compiles itself away, less boilerplate, genuinely pleasant. Smaller
  community.
  - Solid — React-like API, faster runtime. Small community.
  - No framework — doable for something this small, but you'd reinvent the
  state→screen plumbing by hand.

  → React — purely because it's the most documented and AI-assistable, which
  de-risks you (a non-web-dev) + me building it. Note: the typing engine itself
  is plain TypeScript with zero framework dependency, so this choice only
  governs the UI shell, not the core logic.

  3. Build tool / dev server — Vite

  Browsers want one optimized bundle, not your source tree. Vite is the
  toolchain that does it: a dev server with instant hot-reload while coding + a
  production bundler/minifier. It's make + compiler + live-reload, with
  essentially zero config.
  → Vite. No real competitor worth debating in 2026.

  4. Styling — Tailwind vs plain CSS

  CSS is the styling language (colors, layout, animation). Options: write plain
  CSS files, or use Tailwind (utility classes you compose inline — fast to
  iterate, and clean theming via CSS variables, which we want for the theme
  system).
  → Tailwind, but this is the lowest-stakes choice — plain CSS is totally fine
  too.

  5. Drawing the Aegis game — the one genuinely technical choice

  - DOM elements — great for text and UI, but terrible for hundreds of moving
  particles (each is a heavy object).
  - 2D Canvas — an immediate-mode pixel surface, basically a framebuffer you
  draw to each frame. Built into every browser. Perfect for a 2D space shooter
  with explosions/particles.
  - WebGL / PixiJS — GPU-accelerated, handles thousands of sprites. More power,
  more complexity.

  → Hybrid: the typing panel stays DOM (crisp, selectable, accessible text +
  smooth caret); the Aegis battle is a 2D Canvas layer behind/beside it. Only
  upgrade to PixiJS if particle counts ever choke. This is exactly how you'd
  separate "the document" from "the game viewport."

  6. Local storage — IndexedDB (Dexie) vs localStorage

  No server in Phase 1, so persistence lives in the browser:
  - localStorage — tiny synchronous key→string store (~5MB). Fine for settings.
  - IndexedDB — a real embedded, indexed, async database in the browser (think
  SQLite-ish). Right for run history and per-key stats. Dexie is a thin, sane
  wrapper over its clunky native API.

  → Dexie/IndexedDB for stats, localStorage for simple settings.

  7. Hosting — static SPA

  Phase 1 ships no backend: it's just HTML/JS/CSS files; all logic runs
  client-side in the browser. You host the files free on Vercel / Netlify /
  Cloudflare Pages — git push and it's live. A backend only appears in Phase 4
  (accounts/leaderboards).

  # Typesensei

  A typing dojo for programmers: type real source code, fix the muscle memory
  you
  never formally trained, and have a game (Aegis, a reactive space shooter)
  respond
  to clean keystrokes. See `ai-notes/` for the full research + plan.

  ## Stack
  TypeScript + React + Vite + Tailwind. 2D Canvas for the game layer.
  Dexie/IndexedDB
  for local stats. Static SPA — no backend in Phase 1.

  ## Architecture — one rule
  The typing **engine** (`src/engine/`) is pure, framework-agnostic TypeScript.
  It is
  the source of truth and the moat. Everything else subscribes to it:
  - React UI renders the target text and reads engine state.
  - Game skins (Aegis) react to engine events; they NEVER change the text being
  typed.
  Keep `src/engine/` free of React/DOM imports so it stays testable and
  reusable.

  ## Typing model: strict-block
  A wrong character does not advance the cursor. It piles into an "error buffer"
  and
  the run is blocked until backspaced. This makes typo *recovery cost* real and
  measurable — the core metric of this product.

  ## Commands
  - `npm run dev`    — dev server (hot reload)
  - `npm run build`  — production build
  - `npm test`       — vitest (engine unit tests)
  - `npm run lint`   — eslint

  ## Conventions
  - No skipped characters, ever: symbols, spaces, tabs, newlines, backspace all
  count.
  - Never inflate WPM. Honest numbers only.
  - Keep the engine UI-agnostic; put React glue in `src/ui/`.

  The engine — src/engine/types.ts

  export type RunStatus = 'idle' | 'running' | 'done';

  /** One physical key the user produced, in order. The full log lets us 
  recompute
   *  every metric and replay the run. */
  export interface Keystroke {
    key: string;            // the character produced, or 'Backspace'
    expected: string | null; // what the engine expected at the cursor (null for
  backspace)
    correct: boolean;       // true only for a forward key that matched 
  `expected`
    backspace: boolean;     // true if this was a Backspace
    tMs: number;            // ms since the run started (first keystroke = 0)
    cursorAt: number;       // cursor index when the key landed
  }

  export interface EngineState {
    readonly target: string; // raw target text; EVERY char counts 
  (symbols/space/tab/newline)
    cursor: number;          // index of the next character to commit correctly
    errorBuffer: string[];   // wrong chars typed at the cursor, not yet 
  backspaced.
                             // while non-empty the run is "blocked" 
  (strict-block model)
    keystrokes: Keystroke[]; // full ordered log
    startedAt: number | null;
    endedAt: number | null;
    status: RunStatus;
  }

  src/engine/engine.ts — the strict-block state machine

  import type { EngineState, KeyInput } from './types';

  export function createState(target: string): EngineState {
    return { target, cursor: 0, errorBuffer: [], keystrokes: [],
             startedAt: null, endedAt: null, status: 'idle' };
  }

  /** Cursor is parked on an uncorrected typo. The only productive key here is
   *  Backspace — this is what makes a typo actually cost something. */
  export function isBlocked(s: EngineState): boolean {
    return s.errorBuffer.length > 0;
  }

  /** Where the caret renders: after any piled-up wrong characters. */
  export function caretIndex(s: EngineState): number {
    return s.cursor + s.errorBuffer.length;
  }

  /** Pure transition. Returns a NEW state; never mutates the input.
   *  `key` is a single produced character or 'Backspace'; `now` is a ms 
  timestamp. */
  export function applyKey(prev: EngineState, key: string, now: number):
  EngineState {
    if (prev.status === 'done') return prev;

    const startedAt = prev.startedAt ?? now;
    const tMs = now - startedAt;
    const s: EngineState = {
      ...prev, startedAt, status: 'running',
      errorBuffer: [...prev.errorBuffer], keystrokes: [...prev.keystrokes],
    };

    if (key === 'Backspace') {
      if (s.errorBuffer.length > 0) s.errorBuffer.pop();   // recover one wrong 
  char
      else if (s.cursor > 0) s.cursor -= 1;                // un-commit a 
  correct char (allowed)
      s.keystrokes.push({ key, expected: null, correct: false, backspace: true,
  tMs, cursorAt: s.cursor });
      return s;
    }

    const expected = s.target[s.cursor] ?? null;

    if (s.errorBuffer.length === 0 && key === expected) {  // clean forward 
  progress
      s.cursor += 1;
      s.keystrokes.push({ key, expected, correct: true, backspace: false, tMs,
  cursorAt: s.cursor - 1 });
      if (s.cursor >= s.target.length) { s.status = 'done'; s.endedAt = now; }
    } else {                                                // wrong key, or 
  already blocked: pile on
      s.errorBuffer.push(key);
      s.keystrokes.push({ key, expected, correct: false, backspace: false, tMs,
  cursorAt: s.cursor });
    }
    return s;
  }

  export interface RunMetrics {
    durationMs: number;
    wpm: number;          // correct chars / 5 / minutes
    rawWpm: number;       // all forward chars / 5 / minutes
    accuracy: number;     // % correct forward keys of all forward keys
    consistency: number;  // 0–100, steadiness of keystroke rhythm (your "flow" 
  number)
    correctChars: number;
    totalKeystrokes: number;
    typos: number;        // wrong forward keys
    typoCost: number;     // wasted keystrokes = wrong forward keys + backspaces
   ⭐
    perKey: KeyCost[];    // error attribution → heatmap / "which symbol hurts 
  you"
  }

  export function computeMetrics(s: EngineState): RunMetrics {
    const ks = s.keystrokes;
    const forward = ks.filter(k => !k.backspace);
    const backspaces = ks.length - forward.length;
    const correct = forward.filter(k => k.correct);
    const typos = forward.length - correct.length;

    const end = s.endedAt ?? (ks.length ? (s.startedAt! + ks[ks.length - 1].tMs)
  : s.startedAt!);
    const durationMs = s.startedAt == null ? 0 : end - s.startedAt;
    const minutes = durationMs / 60000;

    // Per-key error attribution (only forward keys aimed at a real character).
    const map = new Map<string, KeyCost>();
    for (const k of forward) {
      if (k.expected == null) continue;
      const e = map.get(k.expected) ?? { char: k.expected, attempts: 0, errors:
  0 };
      e.attempts++; if (!k.correct) e.errors++;
      map.set(k.expected, e);
    }

    return {
      durationMs,
      wpm: minutes ? Math.round((correct.length / 5) / minutes) : 0,
      rawWpm: minutes ? Math.round((forward.length / 5) / minutes) : 0,
      accuracy: forward.length ? +(100 * correct.length /
  forward.length).toFixed(1) : 100,
      consistency: computeConsistency(ks.map(k => k.tMs)),
      correctChars: correct.length,
      totalKeystrokes: ks.length,
      typos,
      typoCost: typos + backspaces,
      perKey: [...map.values()].sort((a, b) => b.errors - a.errors),
    };
  }

  /** Steadiness of inter-keystroke intervals → 0–100. Erratic bursts score low,
   *  smooth cadence scores high. This is the metric that maps to "flow". */
  function computeConsistency(times: number[]): number {
    if (times.length < 3) return 100;
    const gaps: number[] = [];
    for (let i = 1; i < times.length; i++) gaps.push(times[i] - times[i - 1]);
    const mean = gaps.reduce((a, b) => a + b, 0) / gaps.length;
    if (mean <= 0) return 100;
    const variance = gaps.reduce((a, g) => a + (g - mean) ** 2, 0) /
  gaps.length;
    const cv = Math.sqrt(variance) / mean;          // coefficient of variation
    return Math.max(0, Math.round((1 - cv) * 100)); // lower variation → higher 
  consistency
  }
